# 성능, 보안, UX를 위한 다음 개선 과제

프로젝트: 관심 필지 지도 (Parcel Explorer)  
작성일: 2026-03-16  
성격: 현행 강제 규칙 문서가 아닌, 현재 코드베이스 기준의 다음 개선 우선순위 제안 문서

## 목적
- 첫 배포 준비 이후 어떤 개선 과제를 우선 투자해야 하는지 성능, 보안, UX 관점에서 정리한다.
- 즉시 실행 규칙은 `docs/engineering-guidelines.md`, 상태 추적은 `docs/TODO.MD`가 담당하고, 본 문서는 그 사이의 우선순위 판단 근거를 제공한다.
- 실제 구현에 착수하는 항목은 `docs/TODO.MD`로 승격해 상태(`todo/doing/blocked/done`)를 관리한다.

## 우선순위 개요
- 현재 코드베이스는 기본적인 보안 통제, 문서 정합성, 첫 배포 준비 수준은 상당 부분 갖춘 상태다.
- 다음 과제는 기초 보완보다 운영 강건성, 확장성 대비, 사용자 체감 품질 개선에 초점을 둔다.
- 우선순위는 구현 비용이 아니라 운영 효과와 리스크 저감 효과를 기준으로 정한다.

## P0 — 운영 안정성/확장 대비
### 1. 공유 상태 기반 레이트리밋 전환
- 로그인 실패 제한, 이벤트 수집, 공개 다운로드 레이트리밋을 인메모리 구현에서 공유 스토어 기반 구현으로 전환한다.
- 현재 단일 인스턴스 전제에서만 안전한 제약을 제거해, 향후 멀티 인스턴스 전환 시 보안/운영 리스크를 줄인다.
- 현재 제한값과 사용자 체감 동작은 유지하되, 상태 저장 위치만 프로세스 외부로 이동하는 것을 기본 방향으로 삼는다.

## P1 — 단기 개선 가치가 큰 과제
### 1. 운영 메트릭 및 알림 기준선 수립
- 요청 지연 시간, 오류율, VWorld API 실패율, 지오메트리 작업 시간/실패 건수, 업로드 거부 사유, 공개 다운로드 사용량을 표준 메트릭으로 수집한다.
- `scripts/run_nonfunctional_checks.py`를 기준으로 성능 기준선을 저장하고, 운영 알림 임계치의 초안을 만든다.

### 2. 로그/이벤트 데이터 보존 자동화
- `map_event_log`, `raw_query_log`, `web_visit_event`의 보존 기간과 정리 정책을 명문화하고 자동화한다.
- SQLite 장기 비대화와 운영 중 조회 성능 저하를 예방하는 것이 목적이다.

### 3. 지오메트리 작업 내구성 고도화
- 현재 재시작 복구는 가능하지만 실행 모델은 여전히 FastAPI 백그라운드 태스크다.
- 업로드 빈도나 데이터 규모가 커질 경우를 대비해, 작업 큐 또는 별도 워커 프로세스로의 이전을 우선 검토한다.

### 4. 공개 지도 로딩 효율 개선
- 현재 프런트는 배치 단위 점진 로딩을 수행하지만, 최종적으로 전체 필지 데이터를 브라우저에 축적한다.
- 다음 단계는 viewport 기반 또는 서버측 필터 기반 조회를 도입해 메모리 사용량과 초기 로딩 부담을 더 낮추는 것이다.

### 5. 관리자 UX 강화
- 업로드 진행 상태, 검증 결과 요약, 지오메트리 재수집 상태, 설정 변경 후 성공/실패 피드백을 더 명확히 보여준다.
- 특히 `SESSION_HTTPS_ONLY`와 같이 재시작이 필요한 설정은 운영자가 놓치지 않도록 경고와 후속 안내를 강화한다.

## P2 — 중장기 UX/운영 완성도 개선
### 1. 공개 UX 검색 흐름 개선
- 빈 결과 상태 메시지 개선
- 점진 로딩 중 상태 표시 강화
- 새로고침/탐색 시 필터 상태 유지
- 모바일 검색/결과 전환 흐름 정교화

### 2. 내보내기/다운로드 거버넌스 강화
- 관리자 export에 대해 더 명시적인 감사 가시성, 범위 제한 옵션, 대량 export 경고를 도입한다.
- 공개 다운로드 파일의 메타데이터나 이력 정보를 관리자 UI에서 확인할 수 있도록 하는 것도 검토한다.

## API 및 운영 영향
- 초기 우선순위 과제 중 공유 상태 기반 레이트리밋과 메트릭 수집은 공개 API 계약 변경 없이 수행하는 것을 기본으로 한다.
- `/api/lands`의 서버측 조회 최적화가 시작되면 `/api/*`와 `/api/v1/*`의 동등성 계약을 함께 재검토해야 한다.
- 새로운 백그라운드 작업 모델을 도입하더라도, 특별한 계약 변경 결정이 없다면 관리자 라우트와 기존 폴링 흐름은 유지하는 것을 우선 원칙으로 삼는다.

## 검증 방향
- 공유 레이트리미터: 여러 앱 프로세스 간 상태 공유, 현재 제한 임계치 유지, `/api`·`/api/v1` 동등성 검증
- 메트릭/보존 정책: 성공/실패 경로 모두에서 계측 누락이 없는지 확인하고, 만료 데이터만 정리되는지 검증
- 서버측 필지 조회: pagination/필터 의미 유지, 부분 데이터 기반 지도 렌더링 정상 동작, 모바일/데스크톱 회귀 확인
- 관리자 UX: 로그인, 업로드, 설정, 통계, 경계선 재수집, 로그아웃 핵심 흐름 회귀 검증
